Auditoría exhaustiva del codebase de Shadow: funcional, bugs, mejoras, refactors, código muerto, documentación, seguridad y tests. Realizada con 6 agentes en paralelo + revisión manual.

## 1. BUGS — Confirmados o Alta Probabilidad

### P0 — Críticos (data loss / corruption risk)

#### 1.0 Concurrent file writes a interactions.jsonl → pérdida de datos
**Archivo:** `src/heartbeat/activities.ts:225-239`
El heartbeat rota `interactions.jsonl` y `conversations.jsonl` con `readFileSync` → filter → `writeFileSync`. Pero hooks externos escriben concurrentemente al mismo fichero. Race condition: heartbeat lee 100 líneas, hook escribe línea 101, heartbeat escribe 50 líneas filtradas → **línea 101 perdida**.

**Fix:** Lock file o escritura atómica (rename pattern).

#### 1.1 Worktrees NUNCA se limpian → leak de disco
**Archivo:** `src/runner/service.ts:95-114`
Se crean worktrees en `.shadow-worktrees/` pero no hay cleanup ni en éxito ni en error. Con el tiempo se acumulan cientos de directorios huérfanos + entradas en `.git/worktrees/`.

**Fix:** Añadir `git worktree remove --force` en el finally de `processRun()`.

### P1 — Altos

#### 1.2 Migrations desordenadas en el array (sin sort)
**Archivo:** `src/storage/migrations.ts`
**Líneas:** 480 (v17), 533 (v15), 551 (v20), 569 (v18)

El array `migrations` tiene versiones fuera de orden: v17 antes de v15, v20 antes de v18. `applyMigrations()` en línea 733 filtra las pendientes pero **no las ordena por versión**:
```typescript
const pendingMigrations = migrations.filter(m => !applied.has(m.version));
```
Actualmente funciona porque las afectadas son independientes, pero **cualquier migration futura fallará si se añade fuera de orden**.

**Fix:** Añadir `.sort((a, b) => a.version - b.version)` después del filter.

#### 1.3 Race condition en daemonShared state
**Archivo:** `src/daemon/runtime.ts:131-132`, `src/daemon/job-handlers.ts:131, 165`
`daemonShared` se muta desde el main loop Y desde job handlers concurrentes sin sincronización. Puede causar heartbeats dobles o perdidos.

#### 1.4 `pendingActivityCount` nunca se resetea
**Archivo:** `src/daemon/runtime.ts:217, 226, 233`
Se incrementa en cada file change y git event pero **nunca se decrementa/resetea**. Crece sin límite hasta restart del daemon.

#### 1.5 detectWorkHours() ignora timezone del usuario
**Archivo:** `src/profile/user-profile.ts:103-111`
Usa `date.getDay()` y `date.getHours()` que operan en timezone del servidor. El profile tiene campo `timezone` pero esta función lo ignora.

#### 1.6 Pagination rota en queries filtradas (MCP tools)
**Archivos:** `src/mcp/tools/observations.ts:41-52`, `src/mcp/tools/suggestions.ts:43-52`
Cuando se filtra por `projectId` o `kind`, se fetchea hardcoded `limit: 100` y se post-filtra en JS. El `total` retornado es `items.length` (post-filtro), no el count real de DB. Pagination rota para >100 items.

#### 1.7 `execSync` con template literal en repo-watcher.ts
**Archivo:** `src/observation/repo-watcher.ts:193`
```typescript
const prevBranch = execSync(`git branch --contains ${previousHead} ...`, {...});
```
Debería usar `execFileSync` como `watcher.ts:tryGitExec()`.

#### 1.8 Timezone frágil en schedules.ts
**Archivo:** `src/daemon/schedules.ts:17-19`
```typescript
function userNow(tz: string): Date {
  return new Date(new Date().toLocaleString('en-US', { timeZone: tz }));
}
```
Anti-pattern que falla cerca de transiciones DST.

#### 1.9 Heartbeat methods exist pero tabla dropped
**Archivo:** `src/storage/database.ts:1098-1145`
`createHeartbeat()`, `getHeartbeat()`, `updateHeartbeat()`, `getLastHeartbeat()`, `listHeartbeats()` — la tabla `heartbeats` fue eliminada en migration v11. Llamar a estos métodos lanza "no such table".

#### 1.10 Trust gate devuelve tool result, no JSON-RPC error
**Archivo:** `src/mcp/server.ts:26-38` + todos los tools
Cuando un trust gate falla, `gate.error` se devuelve como resultado del tool (wrapped en `content`), no como error JSON-RPC. Los clientes no pueden distinguir entre éxito y permiso denegado.

### P2 — Medios

#### 1.11 POST body silenciosamente descartado
**Archivo:** `src/web/server.ts:1049, 1061, 1068`
JSON malformado se convierte silenciosamente en `{}` via `.catch(() => ({}))`.

#### 1.12 Sin bounds checking en pagination (web API)
**Archivo:** `src/web/server.ts` (múltiples endpoints)
`parseInt()` en query params sin validación de rango. `limit=999999999` o `offset=-1` aceptados.

#### 1.13 LIMIT/OFFSET sin Number() en listObservations
**Archivo:** `src/storage/database.ts:902`
```typescript
const pagination = `${filters?.limit != null ? `LIMIT ${filters.limit}` : ''}...`;
```
A diferencia de otros métodos (líneas 699, 1060, 1284) que usan `Number()`, este interpola directo.

#### 1.14 N+1 queries en project detail
**Archivo:** `src/web/server.ts:235-250`
Cada repo, system y contact se fetchea individualmente. Observations y suggestions se filtran en JS tras fetch completo.

#### 1.15 Thought loop se detiene permanentemente en error DB
**Archivo:** `src/daemon/thought.ts:66-74`
Si `ensureProfile()` lanza cualquier error (no solo DB cerrada), el loop de thoughts se para sin retry ni logging.

#### 1.16 `last_fetched_at` se actualiza incluso en error
**Archivo:** `src/observation/remote-sync.ts:65, 74, 87`
Si remote no existe o fetch falla, se marca como "recently fetched", previniendo reintentos.

#### 1.17 Focus mode: 3 implementaciones inconsistentes
**Archivos:** `events/queue.ts:38-42`, `profile/user-profile.ts:221-232`, `heartbeat/state-machine.ts:59-63`
Cada una comprueba focus mode de forma ligeramente diferente (distintos operadores, distintos checks de empty string vs null).

#### 1.18 RRF ftsRank almacena BM25 score, no rank position
**Archivo:** `src/memory/search.ts:122`
`entry.ftsRank = r.rank` guarda el score BM25 en vez del índice de posición en el ranking.

#### 1.19 drainAll no garantiza cleanup
**Archivo:** `src/daemon/job-queue.ts:183-191`
Si el timeout de drain expira, retorna pero los jobs siguen corriendo en background.

#### 1.20 Agent SDK adapter siempre tiene tools vacío
**Archivo:** `src/backend/agent-sdk.ts:32-35`
`tools: []` hardcoded. `pack.allowedTools` aceptado pero nunca usado. Feature incompleta.

## 6. SEGURIDAD

### 6.1 Sin autenticación en :3700
El web server y todos los API endpoints son accesibles sin auth. Cualquiera en la misma red puede leer memorias, observaciones, ejecutar runs, modificar el perfil, triggear jobs.

**Mitigación:** Solo escucha en localhost. Verificar que no haya binding a 0.0.0.0.

### 6.2 Sin CSP headers
Dashboard servido sin Content-Security-Policy. Si algún contenido LLM-generated (markdown de observaciones, memorias, sugerencias) contiene scripts, se ejecutarían.

### 6.3 Trust gates solo en MCP, no en web API
Los endpoints HTTP POST (accept, dismiss, execute, trigger, profile update) no verifican trust level. La protección trust solo aplica a herramientas MCP.

### 6.4 Sin rate limiting
Ni en el web API ni en MCP tools. Un loop runaway de LLM llamando `shadow_check_in` repetidamente inflaría el trust score (+0.3/call).

### 6.5 Markdown rendering
Verificar que `react-markdown` sanitiza output. Si usa plugins que permiten HTML raw, hay riesgo de XSS via contenido LLM.

### 6.6 Path traversal en repo_add
**Archivo:** `src/mcp/tools/entities.ts:153-155`
`resolve(rawPath)` se usa pero no valida que el path sea un git repo real, exista, o esté dentro de bounds permitidos. Trust level 1 permite registrar paths arbitrarios.

### 6.7 `/api/config` expone configuración runtime
**Archivo:** `src/web/server.ts:329-334`
Filtra paths sensibles pero sin verificación localhost-only. Config "safe" puede incluir info sensible.

## 8. MEJORAS SUGERIDAS (no en BACKLOG)

### 8.1 Ordenar migrations por versión
Añadir `.sort()` al filtrado de pending migrations para prevenir problemas futuros.

### 8.2 npm audit
Ejecutar `npm audit` en root y en dashboard. Los proyectos locales acumulan vulnerabilidades de dependencias.

### 8.3 Embedding lazy init race
Si `all-MiniLM-L6-v2` inicializa lazy y dos jobs lo triggean simultáneamente, verificar que no hay double-init o race condition en el singleton.

### 8.4 Force-kill fallback en shutdown
El drain timeout es 60s (`job-queue.ts:183`). Verificar que hay un `process.exit()` de respaldo si el drain no completa — prevenir procesos zombie.

### 8.5 Unified body parser middleware
Crear un helper `parseBody(req, schema?)` que:
1. Lee el body
2. Parsea JSON con error handling
3. Valida con Zod schema si se proporciona
4. Retorna resultado tipado o error HTTP 400

### 8.6 Binding check en web server
Verificar explícitamente que el servidor HTTP escucha solo en `127.0.0.1`, no en `0.0.0.0`.
