## 0. Cómo usar este documento

Cada hallazgo tiene ID único `A<area>-<n>` (ej. `R-01` = runner hallazgo 1). Estructura:

```
### A-XX · Título corto
- **Severidad**: 🔴 crítica | 🟠 alta | 🟡 media | 🔵 baja
- **Tipo**: bug | refactor | mejora | feature | riesgo | perf
- **Fichero:línea**: path:N
- **Estimación**: ~Xh (si es obvio) | - (si requiere diseño)
- **Estado**: ⬜ pending | 🟦 in-progress | ✅ done
- **Depende de**: A-YY (si aplica)
- **Qué**: descripción 1-3 líneas
- **Por qué**: impacto real
- **Fix**: propuesta concreta
```

Para abordar: marcar estado → completar → actualizar `COMPLETED.md` con referencia al ID. No borrar del audit; solo flipear estado.

## 3. Runner + Jobs + Daemon + Hooks

### R-01 · RunQueue phantom adapter (killAll no-op)

- **Severidad**: 🔴 crítica
- **Tipo**: bug
- **Fichero:línea**: `src/runner/queue.ts:81-107`
- **Estimación**: 2h
- **Estado**: ✅ done (2026-04-19) — eliminado phantom, delega a killAllActiveChildren
- **Qué**: `startRun()` crea `new ClaudeCliAdapter(this.config)` y lo guarda en `this.active`. Pero `runner.processRun(run.id)` llama internamente a `selectAdapter(this.config)` y es ESE segundo adapter el que spawnea el proceso real. El adapter guardado en el queue es phantom.
- **Por qué**: `killAll()` itera `this.active.values()` y hace `entry.adapter.kill()` sobre el phantom → no-op. `drainAll(60s)` puede timeout porque el proceso real no recibe señal. Daemon restart puede dejar procesos orfanos. Cubre el caso de stop race que produjo los 162 EADDRINUSE históricos (aunque ese ya está mitigado por plist v2).
- **Fix**: Dos opciones:
  1. Eliminar adapter del queue entirely. Usar `killAllActiveChildren()` (existe en `adapterInstances` global) desde `killAll`.
  2. Refactor `processRun` para aceptar adapter como parámetro y que sea el guardado en queue.
  Opción 1 es menos invasiva.

### R-03 · Summary vs diffStat incoherence (hallucination silenciosa)

- **Severidad**: 🟠 alta
- **Tipo**: bug / riesgo
- **Fichero:línea**: `src/runner/service.ts:328, 372, 393-414`
- **Estimación**: 3h
- **Estado**: ⬜ pending
- **Qué**: `resultSummaryMd` se persiste desde output LLM (línea 328). `diffStat` se computa post-ejecución (línea 372). No hay cruce. LLM puede afirmar "modifiqué 5 archivos" y el diff está vacío.
- **Por qué**: Observación abierta severidad alta con vote. Bloquea confianza en autonomy. Si auto-execute aprueba estos runs, se crean PRs vacíos con resúmenes mentirosos.

### R-04 · pr-sync sin rate limit ni batch

- **Severidad**: 🟠 alta
- **Tipo**: perf / riesgo
- **Fichero:línea**: `src/daemon/handlers/pr-sync.ts:33-38`
- **Estimación**: 2h
- **Estado**: ⬜ pending
- **Qué**: Loop serial `for (const parent of awaitingRuns) { gh pr view }`. Con 20 runs acumulados y gh tardando 1-2s, el handler bloquea 40s.
- **Por qué**: Mientras el handler está bloqueado, job-queue no puede hacer tick; otros jobs queued esperan. Si un repo es privado y gh rate-limitea, no hay backoff.
- **Fix**:
  - Batch `Promise.allSettled` de 5-10 simultáneos.
  - Backoff exponencial si un `gh pr view` falla.
  - Skip temprano si `shared.networkAvailable === false`.
  - Límite de tiempo por batch (ej. 30s máx).
